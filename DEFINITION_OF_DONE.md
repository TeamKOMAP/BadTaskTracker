# Definition of Done (DoD)

## Общие положения

Задача считается завершённой только при выполнении **всех** пунктов данного чек-листа. Это гарантирует качество кода, стабильность приложения и соответствие требованиям.

---

## Чек-лист завершения задачи

### 1. Функциональность
- [ ] Код написан и соответствует требованиям user story
- [ ] Реализована вся описанная в задаче функциональность
- [ ] Обработаны краевые случаи (edge cases)
- [ ] Нет TODO/FIXME комментариев в коде (либо они оформлены как отдельные задачи)

### 2. Качество кода
- [ ] Код соответствует стандартам оформления (C# Coding Conventions)
- [ ] Проведён code review минимум одним членом команды
- [ ] Все замечания review устранены
- [ ] Отсутствуют предупреждения компилятора (warnings)
- [ ] Код проходит статический анализ (если настроен)

### 3. Тестирование
- [ ] Написаны unit-тесты для бизнес-логики (где применимо)
- [ ] Написаны интеграционные тесты для API endpoints (где применимо)
- [ ] Все тесты успешно выполняются (все зелёные)
- [ ] Покрытие кода тестами соответствует минимальным требованиям (если установлены)
- [ ] Функция протестирована вручную (smoke testing)
- [ ] Нет критических багов (Critical/Blocker)

### 4. Документация
- [ ] Обновлен README.md (если добавлены новые endpoints или изменена логика)
- [ ] Добавлены/обновлены XML комментарии в Swagger (summary, params, returns)
- [ ] Обновлена документация по API (если есть отдельная)
- [ ] Добавлены комментарии в коде для сложных алгоритмов

### 5. Интеграция
- [ ] Код слит с основной веткой (develop/helpme) без конфликтов
- [ ] CI/CD pipeline проходит успешно (сборка, тесты)
- [ ] Нет регрессий в существующей функциональности

### 6. QA Acceptance
- [ ] QA инженер проверил задачу
- [ ] Проверены позитивные сценарии (happy path)
- [ ] Проверены негативные сценарии (ошибки, валидация)
- [ ] Проверена интеграция с другими модулями

---

## Типы задач и специфические требования

### Backend (API)
- Unit-тесты для сервисов и бизнес-логики
- Интеграционные тесты для контроллеров
- Валидация входных данных
- Обработка ошибок ( Exception -> HTTP status codes)

### Frontend
- Ручное тестирование UI/UX
- Проверка адаптивности (если применимо)
- Проверка в разных браузерах (Chrome, Firefox)

### База данных
- Миграции протестированы (up/down)
- Схема соответствует моделям
- Индексы добавлены при необходимости

---

## Процесс принятия задачи

```
Разработчик → Code Review → QA Testing → Merge → Done
```

1. **Разработчик** завершает задачу и отмечает пункты DoD
2. **Reviewer** проверяет код и подтверждает выполнение DoD
3. **QA** проводит тестирование и даёт финальное подтверждение
4. **Merge** выполняется только после всех аппрувов

---

## Исключения

В случае невозможности выполнить пункт DoD (например, тесты для legacy кода):
- Согласовать с Tech Lead
- Создать отдельную задачу на доработку
- Документировать причину в комментариях к PR
